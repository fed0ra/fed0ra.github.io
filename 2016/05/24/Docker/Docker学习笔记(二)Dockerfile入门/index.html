    Docker学习笔记(二)Dockerfile入门


    Welcome to Fedor’s blog! If you find any problems, you can tell me in any way.

    一、Dockerfile关键字
    二、Dockerfile最佳实践
    三、docker build自动化构建镜像

    弄明白Docker是怎么回事以后，接下来是如何自建image，有两个方法：

    • 方法一：就是上篇文章中提到的，下载一个官方image，然后在里面run一些你需要的东西，然后通过commit命令把我们在容器的变更持久化下来，并做成一个新的镜像。所以通过commit方式自动化方面很麻烦，所以出现了第二种方法。
    • 方法二：使用Dockerfile。Dockerfile出现是为了解决自动化的问题，就是你执行过的操作的自动化脚本。

    但本质上两个方法是一样的，因为Dockerfile的内容不过是一堆的RUN命令而已，就是在docker容器了写的一些命令（如yum install、export之类），将这些命令做成一种特殊语法的脚本，docker可以解析这种脚本来做成一个新的镜像。
    Dockerfile的文件名一般就叫做Dockerfile，这样比较方便，因为可以直接用来生成image。

    一般使用Dockerfile方法：

    • 启动一个容器，执行要做的那些命令，出问题开始解决，再重新执行，如果这些命令已经没有问题了，优化的已经很好了，然后把这些命令抄到Dockerfile中，最后用docker build去构建。
    • 构建：在Dockerfile文件的目录中执行命令：docker build -t yourname/imagename:yourtagname .
      注意最后是一个英文的句号.当前目录下

    下文中详细讲解docker build使用Dockerfile创建镜像的方法

    一、Dockerfile关键字

    在使用docker build命令根据Dockerfile的内容构建镜像之前，先介绍一下Dockerfile 的基本语法结构，Dockerfile有以下指令选项:

    • FROM
    • MAINTAINER
    • RUN
    • EXPOSE
    • ENV
    • ADD & COPY
    • CMD & ENTRYPOINT
    • VOLUME
    • USER
    • WORKDIR
    • ONBUILD

    FROM

    用法：

    FROM image

    • FROM指定构建镜像的基础源镜像，如果本地没有指定的镜像，则会自动从Docker的公共库pull镜像下来。
    • FROM必须是Dockerfile中非注释行的第一个指令，即一个Dockerfile从FROM语句开始。
    • FROM可以在一个Dockerfile中出现多次，如果有需求在一个Dockerfil 中创建多个镜像。
    • 如果FROM语句没有指定镜像标签，则默认使用latest标签。不建议base image的tag是latest，容易导致每次build都重新pull镜像。

    MAINTAINER

    用法：

    MAINTAINER name

    • 指定创建镜像的用户

    RUN

    两种使用方法：

    RUN
    RUN “executable”, “param1”, “param2”

    • 每条RUN指令将在当前镜像基础上执行指定命令，并提交为新的镜像，后续的RUN都在之前RUN提交后的镜像为基础，镜像是分层的，可以通过一个镜像的任何一个历史提交点来创建，类似源码的版本控制。
    • RUN产生的缓存在下一次构建的时候是不会失效的，会被重用，可以使用–no-cache选项，即docker build –no-cache，如此便不会缓存。
    • 建议合并多个命令，减少层(layer)的数量。docker每RUN一次就是一层，docker的镜像分层(layer)目前能支持的layer数上线是127，超过127层docker就会报错，所以层不要超过127层，而且层越多会导致查一个文件越慢，所以尽量把层尽量控制在非常少的水平线，不建议写很多RUN，建议通过&&把多个命令写到一个RUN下。

    例如：

    #不建议
RUN yum update
RUN yum install vim
RUN touch /etc/my.cnf
#建议
RUN yum update && yum install vim && touch /etc/my.cnf

    EXPOSE

    用法：

    EXPOSE port

    • 告诉Docker服务端容器对外映射的本地端口，需要在docker run的时候使用-p或者-P选项生效。

    ENV

    用法：

    ENV key value # 只能设置一个变量
    ENV key=value … # 允许一次设置多个变量

    • 指定一个环节变量，会被后续RUN指令使用，并在容器运行时保留。

    例如：

    ENV myName="fodor" DIRPATH="/path"
#等同于
ENV myName="fodor"
ENV DIRPATH="/path"

    ADD vs COPY

    用法：

    ADD src… dest
    COPY src… dest

    • 我们要加一个二进制包，或者加一些代码到docker容器内这种情况，这些文件是在我们本地的，这时候用ADD或者COPY命令，来从本地往容器内拷贝。
    • ADD复制本地主机文件、目录或者远程文件URLS从本地添加到容器指定路径中 。
    • COPY复制新文件或者目录从本地并且添加到容器指定路径中 。用法同ADD，唯一的不同是不能指定远程文件 URLS。
    • ADD比COPY多了两个功能：下载URL(通过ftp下载或者其他方式)&解压(ADD一个tar包进去会自动解压)，其他情况两者的使用是一样的。
    • dest目录（目标目录）不存在会自动创建。

    CMD vs ENTRYPOINT

    1.CMD三种使用方法：

    CMD “executable”,”param1”,”param2”
    CMD “param1”,”param2”
    CMD command param1 param2 (shell form)

    • CMD一般放到最下面的一句话，镜像启动变成一个容器以后要执行的命令，这个命令写到CMD里。
    • CMD指定在Dockerfile中只能使用一次，如果有多个，则只有最后一个会生效。
    • CMD的目的是为了在启动容器时提供一个默认的命令执行选项。
    • CMD容易被run指定的命令覆盖。就是如果用户通过docker run命令启动容器时候，指定了一个运行命令会把CMD给覆盖掉。

    注意：CMD会在启动容器的时候执行，build时不执行，而RUN只是在构建镜像的时候执行，后续镜像构建完成之后，启动容器就与RUN无关了，这个初学者容易弄混这个概念，这里简单注解一下。

    2.ENTRYPOINT两种使用方法：

    ENTRYPOINT “executable”, “param1”, “param2”
    ENTRYPOINT command param1 param2 (shell form)

    • 镜像中如果只写一个CMD，会导致别人在执行的时候有可能把指定的CMD覆盖掉，容器想要达到的预期效果就没有了，所以一般推荐用ENTRYPOINT。
    • 配置容器启动后执行的命令，并且不可被docker run提供的参数覆盖，而CMD是可以被覆盖的。如果需要覆盖，则可以使用docker run --entrypoint选项。
    • 每个Dockerfile中只能有一个ENTRYPOINT，当指定多个时，只有最后一个生效。

    复杂逻辑推荐放在ENTRYPOINT中，CMD传入启动参数。
    ENTRYPOINT的时间点优于cmd，晚于run，执行顺序：RUN>ENTRYPOINT>CMD。所以一般会把逻辑写到ENTRYPOINT里，通过CMD传入参数来控制ENTRYPOINT里执行的策略。

    VOLUME

    用法：

    VOLUME [“/data”]

    创建一个可以从本地主机或其他容器挂载的挂载点，后续具体介绍。

    USER

    用法：

    USER daemon

    指定运行容器时的用户名或UID，后续的RUN、ENTRYPOINT、CMD也会使用指定用户。

    WORKDIR

    用法：

    WORKDIR /path/to/workdir

    • 为后续的RUN、CMD、ENTRYPOINT指令配置工作目录。
    • 可以使用多个WORKDIR指令，后续命令如果参数是相对路径，则会基于之前命令指定的路径。

    例：

    WORKDIR /a
WORKDIR b
WORKDIR c
RUN pwd

    最终路径是/a/b/c。
    WORKDIR指令可以在ENV设置变量之后调用环境变量：

    ENV DIRPATH /path
WORKDIR $DIRPATH/$DIRNAME

    最终路径则为 /path/$DIRNAME。

    ONBUILD

    用法：

    ONBUILD [INSTRUCTION]

    • 配置当所创建的镜像作为其它新创建镜像的基础镜像时，所执行的操作指令。

    例：

    [...]
ONBUILD ADD . /app/src
ONBUILD RUN /usr/local/bin/python-build --dir /app/src
[...]

    如果基于image-A创建新的镜像时，新的Dockerfile中使用FROM image-A指定基础镜像时，会自动执行 ONBUILD指令内容，等价于在后面添加了两条指令。

    # Automatically run the following
ADD . /app/src
RUN /usr/local/bin/python-build --dir /app/src

    使用ONBUILD指令的镜像，推荐在标签中注明，例如：nginx:1.7-onbuild。

    二、Dockerfile最佳实践

    使用.dockerignore文件

    类似.gitignore的功能，建议大家在自己的工程中使用一个.dockerignore文件用来明确的指定哪些文件/文件夹不需要一起打包，减少压缩和网络传输的消耗

    避免安装不必要的软件包

    为了降低复杂性、依赖性、文件大小以及构建时间，应该避免安装额外的或不必要的包。例如，不需要在一个数据库镜像中安装一个文本编辑器。

    每个容器都跑一个进程

    尽量做到在大多数情况下，一个容器应该只单独跑一个程序，解耦应用到多个容器使其更容易横向扩展和重用。

    最小化层

    我们知道每执行一个指令，都会有一次镜像的提交，镜像是分层的结构，对于Dockerfile，应该找到可读性和最小化层之间的平衡。前文提到过。

    多行参数排序

    如果可能，RUN时通过字母顺序来排序，这样可以避免安装包的重复并且更容易更新列表，另外可读性也会更强，添加一个空行使用\换行。前文提到过。

    三、docker build自动化构建镜像

    # docker build --help    #运行命令查看帮助

Usage:    docker build [OPTIONS] PATH | URL | -

Build an image from a Dockerfile

  --build-arg=[]                  Set build-time variables
  --cpu-shares=0                  CPU shares (relative weight)
  --cgroup-parent=                Optional parent cgroup for the container
  --cpu-period=0                  Limit the CPU CFS (Completely Fair Scheduler) period
  --cpu-quota=0                   Limit the CPU CFS (Completely Fair Scheduler) quota
  --cpuset-cpus=                  CPUs in which to allow execution (0-3, 0,1)
  --cpuset-mems=                  MEMs in which to allow execution (0-3, 0,1)
  --disable-content-trust=true    Skip image verification
  -f, --file=                     Name of the Dockerfile (Default is 'PATH/Dockerfile')
  --force-rm=false                Always remove intermediate containers
  --help=false                    Print usage
  -m, --memory=                   Memory limit
  --memory-swap=                  Total memory (memory + swap), '-1' to disable swap
  --no-cache=false                Do not use cache when building the image
  --pull=false                    Always attempt to pull a newer version of the image
  -q, --quiet=false               Suppress the verbose output generated by the containers
  --rm=true                       Remove intermediate containers after a successful build
  -t, --tag=                      Repository name (and optionally a tag) for the image
  --ulimit=[]                     Ulimit options

    在Dockerfile文件的目录中执行命令：

    # docker build -t yourname/imagename:yourtagname .

    创建新的image,并命名为yourname/imagename:yourtagname，其中yourtagname是可以省略的，默认是latest。build完成以后会按Dockerfile的内容生成一个新的image。
