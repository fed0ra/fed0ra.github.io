    SaltStack批量管理工具实战（二）


    Welcome to Fedor’s blog! If you found any problems, you can tell me on GitHub.

    一、分组管理
    二、salt模块
    三、文件系统
    四、grains
    五、states
    六、pillar
    七、proxy

    在之前的SaltStack批量管理工具实战（一）文章中，见了，salt的安装部署，使用salt-key命令批量管理服务器，如何迁移salt-master，将返回信息等数据写入数据库，以便后续可视化等。

    随着minion的不断增加和业务的扩展，为了更好的管理minion，我们对它们进行分组。

    一、分组管理

    1.配置例子讲解：

    nodegroups:
  node-01: 'E@CentOS-Server*'
  swmf-app: 'E@swmf-app*'
  swmf-web: 'E@swmf-web*'
  thcb-web: 'E@thcb-web*'
  web: 'N@swmf-web or N@thcb-web'
  tongyong: 'N@swmf-web or N@thcb-web or N@swmf-app or N@node-01'
    • 配置文件中用到YAML语法。
    • nodegroups:固定格式。
    • 第一列swmf-app,node-01等是组名，可以自定义，前面有两个空格。
    • E代表正则匹配，匹配以CentOS-Server开头的所有主机，用引号引起来，引号前有一个空格，下同。
    • 多个组，组合起来形成一个新的组，使用N，or表示加上、与的意思。

    2.分组

    编辑master配置文件：

    # vim /etc/salt/master

    配置如下：

    nodegroups:
  node-01: 'E@CentOS-Server*'

    注：node-01是组名，前面有两个空格，node-01:冒号后有一个空格。

    然后执行命令：

    [root@CentOS-Server01 ~]# salt -N node-01 test.ping
CentOS-Server02.example.com:
    True
CentOS-Server03.example.com:
    True
[root@CentOS-Server01 ~]#

    -N是指定组名，组名不需要加引号

    返回目录

    二、salt模块

    随便指定一台minion，执行sys.doc方法，可以查看使用帮助，命令如下：

    # salt 'CentOS-Server02.example.com' sys.doc

    通过命令，导出文件，制作成文档，使用的时候可自行查看帮助，下文介绍常用的模块。

    # salt 'CentOS-Server02.example.com' sys.doc > ~./salt.txt

    返回目录

    三、文件系统

    用于同步文件，所有文件都要放在/srv/salt（配置文件中默认路径，需要取消注释开启）中，包括salt的sls文件、从salt里面去取的需要同步的文件等。

    1.修改master配置文件

    # vim /etc/salt/master

    取消注释，修改为如下：

    file_roots:
  base:
    - /srv/salt

    必须以file_roots:开头，三个分支，base基础，dev开发，prod生产

    2.创建配置文件指定的目录

    创建上一步配置文件指定的/srv/salt目录，命令如下：

    # mkdir -p /srv/salt
# cd /srv/salt/

    所有走文件系统的都放在/srv/salt/，切换到此目录，创建一个etc用来存放配置文件。

    [root@CentOS-Server01 salt]# mkdir etc
[root@CentOS-Server01 salt]# ll
total 8
drwxr-xr-x 2 root root 4096 Feb 25 19:30 etc
drwxr-xr-x 2 root root 4096 Feb 25 19:30 scripts

    3.编写sls文件

    切换到/srv/salt/etc下，编辑test.sls文件，内容如下：

    [root@CentOS-Server01 salt]# cat /srv/salt/etc/test.sls 
/tmp/hosts:
  file.managed:
    - source: salt://etc/hosts
    - user: root
    - group: root
    - mode: 600
[root@CentOS-Server01 salt]#
    • /tmp/hosts：是name，可以自定义，表示要同步到哪里去。
    • file.managed：表示使用file模块的managed方法管理文件。
    • source：源文件，是要把服务端的哪一个文件同步到minion上去。salt://：代表之前创建的/srv/salt/路径。
    • user：用户
    • group：组
    • mode：同步过去后的权限

    切换到/srv/salt下，编辑top.sls文件，内容如下：

    [root@CentOS-Server01 salt]# cat /srv/salt/top.sls 
base:
  '*':
    - test
[root@CentOS-Server01 salt]# cp /etc/hosts /srv/salt/etc/
    • base:对应三个分支中的base
    • *：代表同步的机器，匹配所有minion机器，让所以机器都执行下一行的文件
    • test：是使用哪一个sls文件，上一步在/srv/salt/etc下创建的test.sls，只写文件名

    master同步

    # salt '*' state.highstate
# salt '*' state.sls test

    两种方法均可，state.sls要指定参数，参数文件在salt文件系统根目录下，也就是/srv/salt下进行查找。
    去minion的/tmp目录下查看，已经同步过去了。

    4.目录结构**

    为了方便管理sls文件，可以在/srv/salt/下建立相关目录用于存放sls文件，这里建立一个名为test-sls的目录用于存放测试用的sls文件（test.sls），执行命令和上面略有区别：

    # salt '*' state.sls test-sls.test

    test-sls.test是文件目录和sls文件名，中间以英文的点分隔。

    有如下目录结构：

    [root@CentOS-Server01 salt]# tree /srv/salt/
/srv/salt/
|-- etc
|   `-- hosts
|-- test-sls
|   |-- init.sls
|   `-- test.sls
`-- top.sls

3 directories, 4 files
[root@CentOS-Server01 salt]#

    如果执行命令，只指定目录，不指定sls文件，如下命令：

    # salt '*' state.sls test-sls

    会默认执行test-sls下名为init.sls的文件，如果没有此文件，会报错。

    5.总结

    • 修改配置文件取消注释开启file_roots
    • 创建目录/srv/salt/
    • /srv/salt/目录下编辑top.sls文件
    • /srv/salt/目录下创建sls目录，进入目录编写sls文件

    返回目录

    四、grains

    用于采集客户端信息，可客户端自定义主动提交，也可服务端定义向下推送，采集完后再汇报上来，pillar相比grains灵活性强一些。

    1.grains

    # salt '*' grains.ls    \\ls列出所有grains名
# salt '*' grains.items    \\列出名和值
# salt '*' grains.item shell    \\取指定的

    2.自定义grains，主要在minion上配置

    首先，修改minion配置文件：

    # vi /etc/salt/minion

    其次，粘贴如下内容：

    grains:
  roles:
    - webserver
    - memcache
  deployment: datacenter4
  cabinet: 13
  cab_u: 14-15

    注意缩进
    grains：语法grains开头
    roles：属性名

    • webserver
    • memcache \\属性名对应的值，下同

    最后，重启minion服务：

    # /etc/init.d/salt-minion restar

    返回目录

    五、states

    所有操作都要基于sls也就是states，用sls来进行批量管理，需要用sls来做。
    sls文件存放一些数据，词典数据、列表、字符串、数字。
    salt状态，其核心是sls，sls表示系统将会是一种什么样的状态，以一种简单的格式来包含这些数据，即配置管理，使用YAML语法。

    YAML：
    缩进，YAML使用固定缩进风格表示数据层结构关系，salt需要每个缩进级别由两个空格组成，禁止使用Tab。
    字典，字典的keys在YAML中以冒号结尾，values在冒号后面，用一个空格分隔。
    列表项，使用一个短横杠-后加一个空格，同一个列表用同样的缩进级别。

    states模块：
    states可以参考官方文档

    例子：
    使用sls安装nginx，管理nginx配置文件，当nginx配置文件被修改后，自动更新配置文件，并重启nginx。

    # mkdir -p /srv/salt/etc/nginx/conf.d
# mkdir -p /srv/salt/nginx
# cp /etc/nginx/nginx.conf /srv/salt/etc/nginx/
# cp /etc/nginx/conf.d/default.conf /srv/salt/etc/nginx/conf.d/
# cd /srv/salt/nginx/
# vim init.sls

    nginx的init.sls文件内容

    nginx:
  pkg:
    - installed
  service:
    - running
    - enable: True
    - reload: True
    - watch:
      - pkg: nginx
      - file: /etc/nginx/nginx.conf
      - file: /etc/nginx/conf.d/default.conf
/etc/nginx/nginx.conf:
  file.managed:
    - source: salt://etc/nginx/nginx.conf
    - user: root
    - group: root
    - mode: 644
/etc/nginx/conf.d/default.conf:
  file.managed:
    - source: salt://etc/nginx/conf.d/default.conf
    - user: root
    - group: root
    - mode: 644

    两种方法执行定时任务，一种是使用salt-call，二中是使用pillar。
    minion端执行命令：

    # salt-call state.sls nginx

    等待命令执行完毕，查看是否安装启动成功，端口是否存在：

    # salt '*' cmd.run 'rpm -qa | grep nginx'
# salt '*' cmd.run 'netstat -lntup | grep 80'

    修改/srv/salt/etc/nginx/conf.d/default.conf改变nginx端口，再执行stal-call，查看端口是否改变。

    定时任务：每五分钟同步一次配置文件只需在minion端将salt-call state.sls nginx命令加入crontab定时任务

    返回目录

    六、pillar

    1.pillar定时执行示例

    schedule:
  highstate:
    function:state.highstate
    minutes:1

    pillar可参考pillar的官方文档

    2.SaltStack定时批量更新所有nginx服务配置

    去编辑master的配置文件，去掉注释开启pillar模块：

    # vim /etc/salt/master
    pillar_roots:
      base:
        - /srv/pillar

    创建pillar目录，所有的pillar相关都写在这里

    # mkdir -p /srv/pillar

    编辑top,pillar中top必须写

    base:
  '*':
    - nginx
    • base:file_root中的base
    • *：匹配主机名

    创建nginx目录，编辑nginx.sls

    # cd /etc/salt/
# mkdir nginx
# cd nginx/
# vim nginx.sls

    nginx.sls内容如下：

    [root@CentOS-Server01 nginx]# cat init.sls 
schedule:
  nginx:
    function: state.sls
    minutes: 1
    args:
      - 'nginx'

    说明：

    • nginx：nginx的名字，好识别。
    • function：执行一个函数state.sls。
    • minutes：每分钟执行一次。
    • args：参数为nginx。

    刷新指定minion的pillar.data数据，并查看pillar

    # salt '*' saltutil.refresh_pillar
# salt '*' pillar.data

    返回目录

    七、proxy

    ####测试环境：

    角色:域名:IP
master:CentOS-Server01.example.com:192.168.174.10
minion01:CentOS-Server02.example.com:192.168.174.12
minion02:CentOS-Server03.example.com:192.168.174.13
proxy01:CentOS-Server04.example.com:192.168.174.14
proxy02:CentOS-Server05.example.com:192.168.174.15

        -- proxy01 -- minion01,更多

master    -- proxyxx -- minionxx...

        -- proxy02 -- minion02,更多

    1.使用proxy代理

    minion越来越多，master撑不住，要进行分组分布式的管理，就用到了proxy

    2.syndic

    proxy利用syndic工具。salt Syndic接口是一个工具，允许建立salt命令拓扑结构。

    3.过程原理

    master发出一条命令，命令原封不动的给proxy，proxy接到命令，proxy会匹配minion的id，发给minion，minion把结果发给proxy，proxy返回给master，最后master上写一个自定义脚本，数据写入mysql。

    4.安装syndic

    syndic代理基于master，所以既需要安装salt-syndic代理，还需要安装salt-master。
    在proxy01和proxy02上操作：

    # yum install salt-syndic salt-master -y

    修改proxy端的配置文件

    # vi /etc/salt/master

    取消syndic_master前的注释，值改成master的ip

    syndic_master: 192.168.174.10

#syndic_master：更高级别master的ip地址
#syndic_log_file：syndic日志文件路径，绝对或相对路径。
#order_masters:True    #不加此参数，不进行转发。

    代理和master都要启动

    # /etc/init.d/salt-syndic start
# /etc/init.d/salt-master status

    回到master上查看添加的key，添加proxy01和proxy02的key。 

    [root@CentOS-Server01 nginx]# salt-key -L
Accepted Keys:
CentOS-Server02.example.com
CentOS-Server03.example.com
Denied Keys:
Unaccepted Keys:
CentOS-Server04.example.com
CentOS-Server05.example.com
Rejected Keys:
[root@CentOS-Server01 nginx]# salt-key -A -y
The following keys are going to be accepted:
Unaccepted Keys:
CentOS-Server04.example.com
CentOS-Server05.example.com
Key for minion CentOS-Server04.example.com accepted.
Key for minion CentOS-Server05.example.com accepted.
[root@CentOS-Server01 nginx]# salt-key -L
Accepted Keys:
CentOS-Server02.example.com
CentOS-Server03.example.com
CentOS-Server04.example.com
CentOS-Server05.example.com
Denied Keys:
Unaccepted Keys:
Rejected Keys:
[root@CentOS-Server01 nginx]#

    修改minion的hosts解析salt为对应的proxy，minion01解析proxy01的地址，minion02解析proxy02的地址。

    # salt 'CentOS-Server02.example.com' cmd.run "cat /etc/hosts | grep salt |sed -i 's/192.168.174.10/192.168.174.14/' /etc/hosts"
# salt 'CentOS-Server03.example.com' cmd.run "cat /etc/hosts | grep salt |sed -i 's/192.168.174.10/192.168.174.15/' /etc/hosts"

    master上也需要踢掉minion，此时的minion不归master管辖，proxy归master管辖。 

    [root@CentOS-Server01 nginx]# salt-key -d CentOS-Server02.example.com -y

CentOS-Server02.example.com

CentOS-Server03.example.com

[root@CentOS-Server01 nginx]#

    master,proxy上执行test.ping

    # salt '*' test.ping
