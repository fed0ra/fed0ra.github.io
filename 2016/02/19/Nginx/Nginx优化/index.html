    1.隐藏nginx header里版本号信息

    查看nginx版本，执行如下命令

    # curl -I -s 192.168.174.10 | grep Server
# curl -I 192.168.174.10 2> /dev/null | grep Server

    编辑nginx配置文件在http标签加server_tokens off
    重启nginx

    nginx -t
ngins -s reload

    2.更改源码隐藏软件名称及版本号

    切换到nginx源码下

    # cd /usr/local/src/nginx-1.7.0/src/http/

    a) 编辑ngx_http_special_response.c

    # vim ngx_http_special_response.c

    修改为：

    28 static u_char ngx_http_error_tail[] =
29 "<hr><center>apache</center>" CRLF

    b) 编辑ngx_http_header_filter_module.c

    # vim ngx_http_header_filter_module.c

    修改为：

    49 static char ngx_http_server_string[] = "Server: Apache" CRLF;
50 static char ngx_http_server_full_string[] = "Server:Apache " NGINX_VER CRLF;

    或使用以下命令：

    # grep 'Server: nginx' ngx_http_header_filter_module.c
# sed -i 's#Server: nginx#Server: apache#' ngx_http_header_filter_module.c

    c) 如果之前安装了nginx，查看nginx编译参数，重新编译，命令如下（没装过nginx编译安装）

    # /usr/local/nginx/sbin/nginx -V
# ./configure --prefix=/usr/local/nginx --user=www --group=www --with-http_ssl_module --with-http_flv_module --with-http_stub_status_module --with-http_gzip_static_module
# /etc/init.d/nginx stop
# make && make install
# /etc/init.d/nginx start
# curl -I -s 192.168.174.10 | grep Server

    3.更改nginx默认的用户和用户组

    查看nginx默认用户

    # grep '#user' nginx.conf.default

    创建用户

    # useradd -s /sbin/nologin -g www -M www

    编译时候需指定–user=www –group=www
    编辑nginx配置文件在main标签修改user参数为www

    4.配置nginx worker进程数

    worker进程数大小按照CPU核数来确定，最好和CPU核数一致，先查看CPU合数：

    # grep 'physical id' /proc/cpuinfo

    或使用top命令，后按数字键1
    修改使用如下命令：

    # sed -i 's@worker_processes  1@worker_processes  2@g' /usr/local/nginx/conf/nginx.conf

    5.根据CPU核数进行nginx进程优化

    为了避免资源分配不均

    编辑nginx配置文件在main标签添加：
    四核CPU参数配置示例：

    worker_cpu_affinity 0001 0010 0100 1000;

    八核CPU参数配置方法一示例：

    worker_cpu_affinity 00000001 00000010 00000100 00001000 00010000 00100000 01000000 10000000；

    八核CPU参数配置方法二示例：

    worker_cpu_affinity 0001 0010 0100 1000 0001 0010 0100 1000；

    通过webbench压力测试，看CPU情况

    # webbench -c 20000 -t 180 http://192.168.174.10/

    另一种优化方法是用taskset，taskset让进程运行在指定的CPU，例

    # taskset -c 1,2,3 /etc/init.d/mysqld start

    taskset的使用请参考edwardlost’ blog

    以下给出一些配置实例：

    1、2核CPU，开启2个进程
worker_processes  2;  
worker_cpu_affinity 01 10;

2、2核CPU，开启4进程
worker_processes 4;
worker_cpu_affinity 01 10 01 10;

3、2核CPU，开启8进程
worker_processes  8;  
worker_cpu_affinity 01 10 01 10 01 10 01 10;

4、8核CPU，开启2进程
worker_processes  2;  
worker_cpu_affinity 10101010 01010101;

    说明：10101010表示开启了第2,4,6,8内核，01010101表示开始了1,3,5,7内核

    编辑nginx配置文件，在events标签中添加use epoll;，修改后如下：

    events{

}

    7.调整nginx单个进程允许的客户端最大连接数

    编辑nginx配置文件，在events标签中添加worker_connections，修改后如下：

    events {
    worker_connections  1024;
    #动态8192、4096已经很大了，三万已经很大了。
}

    8.nginx每个进程打开的最大连接数(文件数)

    编辑nginx配置文件在main标签添加：

    worker_rlimit_nofile 65535;

    9.优化服务器名字hash表的大小

    编辑nginx配置文件在http标签添加：

    server_names_hash_bucket_size 64;
server_names_hash_max_size 512;

    10.开启高效的文件传输模式

    编辑nginx配置文件在http, server, location标签添加：

    sendfile        on;
tcp_nopush on | off;

    11.设置连接的超时时间

    编辑nginx配置文件在http, server, location标签添加：

    keepalive_timeout  65;
tcp_nodelay on;    #包含keepalve参数才有效
client_header_timeout 15;
client_body_timeout 15;
send_timeout 15;

    php网站建议短连接，java网站建议连接

    12.上传文件大小限制(动态应用)

    编辑nginx配置文件在http, server, location标签添加：

    client_max_body_size 10m;

    13.fastcgi调优(配合PHP引擎动态服务)

    编辑nginx配置文件在http, server, location标签添加：

    fastcgi_connect_timeout 60;#指定连接到后端fastcgi的超时时间
fastcgi_send_timeout 60;#想fastcgi传送请求的超时时间，这个值指定已经完成两次握手后向fastcgi传送请求的超时时间
fastcgi_read_timeout 60;#指定接收fastcgi应答的超时时间，这个值指定已经完成两次握手后接收fastcgi应答的超时时间
fastcgi_buffer_size 64k;#指定读取fastcgi应答一部分需要用多大的缓存区，这个值表示使用1个64KB的缓冲区读取应答的第一部分(应答头)可以设置为fastcgi_buffers选项指定的缓冲区大小
fastcgi_buffers 4 64k;#4个64k的缓冲区
fastcgi_busy_buffers_size 128k;#建议为fastcgi_buffers的两倍
fastcgi_temp_file_write_size 128k;#在写入fastcgi_temp_path是将用多大的数据块，默认值是fastcgi_buffers的两倍，设置上述数值设置太小时若负载上来时可能报502 Bad Gateway
fastcgi_cache mean_nginx;#表示开启fastcgi缓存并为其指定一个名称，开启缓存非常有用，可以有效降低cpu负载，并且防止502错误发生，但是开启缓存也可能会引起其他问题，根据具体情况选择
fastcgi_cache_valid 200 302 1h;#用来指定应答代码的缓存时间，实例中的值表示将200和302应答缓存1小时
fastcgi_cache_valid 301     1d;#将301应答缓存1天
fastcgi_cache_valid any      1m;#将其他应答缓存为1分钟
fastcgi_cache_min_uses 1;#缓存fastcgi_cache_path指令inactive参数值时间内的最少使用次数

    14.压缩

    优势：节省带宽；加快打开速度，提升用户体验。
    劣势：消耗CPU,MEM。

    如何使用压缩：首先分析，哪些资源压缩比高，占用资源少。其次，文本文件(js,css,html)需要压缩

    压缩功能同apache的mod_deflate压缩功能，依赖nginx_http_gzip_module模块，默认已安装。
    配置及解释如下如下：

    gzip on;
gzip_min_length 5k;#设置允许压缩的页面最小字节数，页面字节数从header同的Content-Length中获取。默认是0，不管页面多大都压缩。建议设置成大于1k。如果小于1K可能会越压越大。
gzip_buffers 4 16k;#压缩缓冲区大小。表示申请4个单位为16k的内存作为压缩结果流缓存，默认值是申请与原始数据大小相同的内存空间来存储gzip压缩结果。
#gzip_http_version 1.0;#压缩版本(默认1.1，前端为squid2.5时使用1.0)用于设置识别协议版本，默认1.1，目前大部分浏览器已经支持GZIP解压，使用默认即可。
gzip_comp_level 2;#压缩比率。用来指定GZIP压缩比，1压缩比最小，处理速度最快，9压缩比最大，传输速度快，但处理最慢，也比较消耗cpu资源。
gzip_types text/html/plain application/x-javascript text/css application/xml;
gzip_types text/css text/xml application/javascript;#指定压缩的类型，“text/html”类型总是会被压缩。
gzip_vary on;#vary header支持。该选项可以让前段的缓存服务器经过GZIP压缩的页面，例如用squid缓存经过nginx压缩的数据。

    完整配置如下：

    gzip  on;
gzip_min_length 1k;
gzip_buffers 4 32k;
gzip_http_version 1.1;
gzip_comp_level 9;
#gzip_types text/html/plain application/x-javascript text/css application/xml;
gzip_types text/css text/xml application/javascript;
gzip_vary on;

    提示：不同的nginx版本压缩类型有改变
    如何查看压缩类型？

    # cd /usr/local/nginx/conf/
# less mime.types

    最后我们使用火狐的slow插件来查看压缩情况。

    15.缓存

    • expires是网站元素在客户端过期的设置。
    • 作用：可以把大量的资源的文件存放到客户端，超过expires设定的时间久过期。在expires过期时间内，用户再访问网站，已经缓存的内容，就不会去网站下载了。
    • 优势：节省带宽，相同的东西客户端只下载一次。浏览加快，提升用户体验，很多图片文件在本地缓存了。减少服务器压力，支持并发高了。
    • 劣势：网站更新内容后，客户端还是旧的。测试的时候，希望是新的，总是看不到。广告，统计的代码，就不准确了。
    • 解决劣势：根据业务需求设置expires时间。图片更改(重新上传)1年–10年。广告，统计，不缓存。
    • 网站改版元素要改名，js，css
    • 缓存的cdn的资源怎么控制，首先清理源站图片，然后通过cdn提供的接口(api或web界面)发给cdn，让它清理对应缓存。

    配置如下：

    #根据扩展名控制
location ~ .*\.(gifp|jpg|jpeg|png|bmp|swf)$ {
    expires    365d;
}

location ~ .*\.(js|css)$ {
    expires    30d;
}
#根据目录进行控制
location ~ ^/(images|javascript|js|css|flash|media|static)/ {
    expires    360d;
}
#根据单个文件控制
location ~ (robots.txt) {
log_not_found off;
    expires    7d;
break;
}
#为robots.txt为7天并不记录404错误

    robots.txt设置可以或禁止各种搜索引擎访问的资源，其缺点是暴露目录结构。

    16.user-agent介绍及nginx服务防爬虫

    User-agent: Baiduspider
Disallow: /

User-agent: baiduspider
Disallow: /

    假定一个场景：某个网站它可能不希望被网络爬虫抓取，例如测试环境不希望被抓取，以免对用户造成误导，那么需要在该网站中申明，本站不希望被抓取。有如下方法：

    方法一：修改nginx.conf，禁止网络爬虫的ua，返回403。

    server { 
listen 80; 
server_name 127.0.0.1; 
#添加如下内容即可防止爬虫
if ($http_user_agent ~* "qihoobot|Baiduspider|Googlebot|Googlebot-Mobile|Googlebot-Image|Mediapartners-Google|Adsbot-Google|Feedfetcher-Google|Yahoo! Slurp|Yahoo! Slurp China|YoudaoBot|Sosospider|Sogou spider|Sogou web spider|MSNBot|ia_archiver|Tomato Bot") 
{ 
return 403; 
}

    方法2：网站更目录下增加Robots.txt，放在站点根目录下。

    通过robots文件生成，站点可以针对现在的搜索引擎按照想要的规则生成robots.txt文件。

    知识扩展：
    　　robots.txt是搜索引擎中访问网站的时候要查看的第一个文件。robots.txt文件告诉蜘蛛程序在服务器上什么文件是可以被查看的。
    　　当一个搜索蜘蛛访问一个站点时，它会首先检查该站点根目录下是否存在robots.txt，如果存在，搜索机器人就会按照该文件中的内容来确定访问的范围；如果该文件不存在，所有的搜索蜘蛛将能够访问网站上所有没有被口令保护的页面。百度官方建议，仅当您的网站包含不希望被搜索引擎收录的内容时，才需要使用robots.txt文件。如果您希望搜索引擎收录网站上所有内容，请勿建立robots.txt文件。
    　　Robots协议是国际互联网界通行的道德规范，基于以下原则建立：
    　　　　1、搜索技术应服务于人类，同时尊重信息提供者的意愿，并维护其隐私权；
    　　　　2、网站有义务保护其使用者的个人信息和隐私不被侵犯。
    　　当然，如果搜索引擎不遵守约定的Robots协议，那么通过在网站下增加robots.txt也是不起作用的。（在正式环境中，可以适当允许搜索引擎抓取收录）

    禁止不同浏览器软件访问：

    if ($http_use########################r_agent ~* "Firefox|MSIE")
{
    return 403;
}

    17.nginx日志优化相关

    略
    日志切割脚本nginx_log_split.sh

    #!/bin/bash
#Date:          2013-01-28
#Author:        Created by Fedor
#Mail:
#Version:       1.0
#Function:      This script is cut nginx log 
####This script run at 00:00
####The Nginx logs path
####clean log's  15 days ago

#source function library
#. /etc/init.d/functions

logs_path="/usr/local/nginx/logs"
logs_path1="/opt/data/nginx/access_log"
nginx_pid="/usr/local/nginx/logs/nginx.pid"
host_ip="xx.xx.xx.xx"

#create log directory
mkdir -p ${logs_path1}/$(date -d "yesterday" +"%Y%m%d")/

#move log to new directory
mv ${logs_path}/access.log ${logs_path1}/$(date -d "yesterday" +"%Y%m%d")/$host_ip-access.log
mv ${logs_path}/error.log ${logs_path1}/$(date -d "yesterday" +"%Y%m%d")/$host_ip-error.log

#make new log
kill -USR1 `cat ${nginx_pid}`

    将脚本添加到计划任务crontab -e：

    #cut nginx access log by mean at 20150601
00 00 * * * /bin/sh /server/scripts/nginx_log_split.sh > /dev/null 2>&1
00 00 * * * /usr/bin/find /opt/data/nginx/access_log/ -type d -mtime +15 | xargs rm -rf

    18.最小化nginx目录及文件权限设置

    安全站点目录权限：

    • 站点内所有文件和目录的用户和组都应为root。
    • 站点内所有目录权限为755。不能往目录放文件
    • 站点内所有文件权限是默认的644。不能改文件
      注意：网站服务的程序使用的用户不能用root

    以上权限设置可以做到防止黑客上传木马，以及修改站点文件，但是，合理的用户上传的内容也被拒之门外。
    如何解决可以让合法的用户传文件又不至于被黑客利用攻击
    解决方案：

    • 大多数网站做法chown -R nginx.nginx /data（站点目录），比较危险。
    • 所有站点777，最危险的???
    • 找到上传目录，授权nginx可以访问，比较安全的做法。
    • 架构上解决。
    • mount安全优化或者放到/etc/fstab。mount -o nosuid,noexec,nodev

    19.nginx站点目录及文件URL访问控制

    nginx配置限制指定目录下的php程序被解析

    location ~ ^/images/.*\.(php|php5)$ {
    deny all;
}
location ~ ^/static/.*\.(php|php5)$ {
    deny all;
}
location ~* ^/data/(attachment|avatar)/.*\.(php|php5)$ {
    deny all;
}

#这些目录限制必须要配置在一下配置的前面
location ~ .*\.(php|php5)?$ {
    #root           html;
    fastcgi_pass   127.0.0.1:9000;
    fastcgi_index  index.php;
    #fastcgi_param  SCRIPT_FILENAME  /scripts$fastcgi_script_name;
    include        fastcgi_params;
}

#禁止访问*.txt文件
location ~* \.(txt|doc)$ {
    if (-f $request_filename) {
root /data/www/www
#rewrite ……可以重定向到某个URL
break;
}
}


#禁止访问指定目录下所有的文件和目录

#单目录
location ~ ^/(static)/ {
    deny all;
}
location ~ ^/static {
    deny all;
}
#多目录
location ~ ^/(static|js) {
    deny all;
}

#禁止访问目录并返回指定的http状态码
location /admin/ { return 403; }
