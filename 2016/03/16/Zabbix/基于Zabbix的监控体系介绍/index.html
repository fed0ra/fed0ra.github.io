    一、硬件监控

    定期对机房巡检，比如机房巡检所有服务器各种状态，哪些电源线、网线松动，或者硬盘闪黄灯等。

    从服务器本身考虑，一般个大硬件厂商都提供了远程控制口，即服务器上的远程控制卡，如ibm的imm。远程控制卡是完全独立的，可以通过它来控制服务器开关机，硬盘故障也可以通过远程控制卡看日志，购买服务器的时候建议安装，能减少工作量。

    在linux下做硬件监控，我们通过使用ipmitool实现linux系统下对服务器的IPMI管理。

    IPMI即智能平台管理界面（IPMI，Intelligent Platform Management Interface)是管理基于 Intel® 结构的企业系统中所使用的外围设备采用的一种工业标准，用户可以利用IPMI监视服务器的物理健康特征，如温度、电压、风扇工作状态、电源状态等。

    使用IPMI管理先决条件：

    • 1.硬件支持ipmi协议，现在一般服务器都支持
    • 2.操作系统支持，linux通过内核提供了对ipmi的支持

    • 3.用户管理工具，linux下有ipmitool，可以通过

      # yum install OpenIPMI ipmitool -y
# lsmod | grep ipmi  # 查看没有加载ipmi模块，需要启动ipmi
# /etc/init.d/ipmi start
# lsmod | grep ipmi
# ipmitool help;

    可以使用IPMI来监控，通过ipmitool命令过滤出需要的信息，超过阀值之后报警。

    IPMI命令执行两种方式，本地和网路，所以还需要给远程控制卡设置IP地址，一般是在bios中给远程控制卡设置IP，像戴尔的也可以在面板上设置ipmi的IP，还可以设置impi的overline，即不用单独插一个网线直接用物理机的网线。但最好给远程控制卡单独插一根网线，单独设置一个IP地址，维护起来方便。具体操作略。

    IPMI缺点，获取不到硬盘的状态，硬盘状态我们可以使用MegaCli工具，可以查看raid磁盘阵列状态。
    参考：http://www.ttlsa.com/tools/megacli-tool-query-raid-status/

    二、系统监控

    并不是开源的一定最好，有时候花些钱比做一个方案更有效更直接，对业务带来的价值更多。
    架构和流程设计根据业务来做。

    Linux系统级别监控一般需要监控cpu、内存、硬盘、网络。

    做监控首先要了解将空对象的运行原理，其次，要知道性能的基准线，知道监控对象的指标达到什么时候是好，什么时候是坏，最后，要知道怎么做，做监控方法千千万，没有一种方法是完全对的。

    常用监控工具sysstat：

    # yum install sysstat -y

    1. CPU监控

    查看CPU信息两个命令：

    # cat /proc/cpuinfo
# lscpu

    CPU指标：
    上下文切换：同一时间只有一个进程在CPU运行，CPU会给每个进程分配时间片，当某一进程时间片结束时，有可能让其他进程运行，或者被高优先级的进程所抢占，所以CPU的作用，是做时间片的分享、做调度。CPU为了调度执行的进程，会把相关数据从CPU寄存器中拿出来，把另一个进程的信息拿来载入到CPU寄存器中，这个操作叫做上下文切换，上下文切换越多系统开销越大，但一般不会出现。
    上下文切换怎样判断好坏，要根据当前cpu繁忙程度确定，cpu越繁忙上下文切换越高，也可以通过图形看，如果cpu不繁忙上下文切换高，肯定有问题，还应该确定服务的类型来确定基准线。另外，服务有io密集型的，有cpu密集型的，简单来说有些服务占用cpu，有些服务占io。计算类吃cpu，数据库占io，数据库一般占io，如果数据库cpu达到60%，可能数据库慢查询一堆了。

    负载：通过uptime命令显示cpu负载，命令会显示cpu的1/5/15分钟的平均负载，此数据是从cat /proc/loadavg文件读来的。
    通常会通过负载判断系统运行状态，一般情况下，每个cpu的运行队列小于等于3是正常的，如果两个核，负载小于等于6在可以接受范围内，超过六代表负载比较高了。

    cpu使用率：通过top或者mpstat 1命令查看CPU使用率。CPU使用率中会有两种状态，用户态和内核态，普通的程序进程默认运行在用户态，当做系统调用的时候运行内核态，例如open一个文件，这就是一个系统调用，会在内核态打开文件，这样做是为了隔离保证系统安全性，普通程序没有权限直接操作硬件，做任何操作都要将用户态切换到内核态来执行。
    CPU使用率，通常下，内核态和用户态的比例是35%-30%和65%-70%，如果内核态70%，用户态30%，可能有问题，证明程不停的在做系统调用，正常的运行给耽误住了。

    2. 内存监控

    # vmstat 1

    监控内存使用率，使用多少，剩余内存多少等，看内存一般看-/+ buffers/cache，

    三、网络监控

    网路一般监控网卡流量，比如流量从哪个ip过来的，监控工具iftop安装：

    # yum install iftop -y;iftop

    1. nmon工具

    可以提供更详细信息，需掌握，比如做性能报表，nmon可列出比较详细的性能指标，实践操作略。

    2. 网络监控

    可以使用第三方的系统，如监控宝、基调、博瑞等，监测机房质量可以通过站长工具超级ping检测机房网络质量等。

    开源工具使用smokeping，是一款网络状态监控工具。支持分布式，可配置多个slave，全国十个机房，每个机房装一个slave，来交叉ping另外的ip，检测另外九个机房到本机房的延时，可以将检测到的网络延时详细的显示出来，节点越多越准确，因为用户访问我们和我们访问用户路由是不一样的。如果只有一个机房，不能交叉ping，就从机房ping全国各地的，可以间接反应出自己的网络状况，但是不准确，因为过去和过来路由不同，过去没问题，过来也许就有问题。

    所以做网络监控，要么第三方监控宝，要么使用开源工具（smokeping等）。

    四、应用监控

    服务器上通常会运行很多服务，这些服务也需要监控，介绍最常用的服务监控，如apache，nginx这些web服务，memcached，redis等等。

    1. Apache

    Apache有一个mod_status的模块，用来做Apache相关的性能指标监控。

    使用Apachemod_status模块：

    • 编译Apache时候，将此模块编译进去
    • 在配置文件中load此模块
    • 写相关配置文件
    • 实践操作略

    2. Nginx

    Nginx有一个status状态监控模块。

    使用Nginx状态监控模块：

    • 编译Nginx状态监控模块
    • 在Nginx配置文件中加配置，实践操作略，可在博客其他文章中找到具体操作

    3. Memcached和Redis

    使用以下命令，可以列出Memcached和redis一些状态：

    #Memcached状态，可以列出命中率，线程等一系列指标
# telnet 192.168.174.12 11211 stats

#Redis状态，版本号客户端连接数等一些列指标
# redis info

    基本所有的应用程序都会提供相关的接口，可以获取其运行状态等。

    获取应用运行状态后，我们可以通过脚本过滤出需要监控的指标，报警等等。

    另外，项目上线前期，运维应该考虑的

    • 应用提供接口，能让运维看到，以便运维以后监控来使用。
    • 安全，怎么实现acl控制，内部白名单机制等。例如，测试环境有可能连接生产环境，会产生安全问题，如果是http请求，可以在前面放一个nginx，设置一个acl控制，如果是tcp，可以使用iptables规则等。生产只允许生产ip段访问，不允许测试ip段访问。
    • 验证功能，内部接口要求开发有验证功能，用户名密码等等方式，技术某些情况下测试环境连接到生产环境也可以避免不必要的问题出现。

    Zabbix的安装使用，见博客Zabbix标签下文章。

    六、流量监控和监控可视化

    1. 流量监控分析(重要)

    流量分析可以把我们的业务数据和监控数据对应起来。

    做流量监控我们可以使用第三方的，如百度统计、CNZZ统计、google分析等工具，实现统计网站PV、UV、趋势分析、来源分析、页面分析、访客分析等等。

    不想把数据放在个大统计，我们可以使用开源流量分析工具，如piwik

    Piwik是一个PHP和MySQL的开放源代码的Web统计软件。它给你一些关于你的网站的实用统计报告，比如网页浏览人数, 访问最多的页面, 搜索引擎关键词等等。Piwik有甚多不同功能的插件，可以添加或移除不需要的功能，Piwik同样可以安装在服务器上面，这样数据就保存在服务器上面。可以非常容易的插入统计图表到网站或是后台的控制面板中。安装完成后，只需将一小段代码放到将要统计的网页中即可。

    piwik官网https://piwik.org/，实践操作略。

    2. 监控可视化(关系网)

    把所有监控项之间建立关系网，业务、网络、系统、数据库之间都是存在关联的，找到它们之间关联，像百度的ECharts(http://echarts.baidu.com/)

    Zabbix也可是实现简单的关系网，把相关联的数据图表做成一个screen，即放到一个大的图表上，某一项数据一变动，可以看到所有我们自己定义的关联图表的变动。操作实现略。

    七、其他监控

    • 页面性能监控，一般第三方博瑞基调提供监控数据，如DNS解析时间、传输时间等等。
    • 安全监控，nginx+lua实现wab应用防火墙等。
    • ELK日志监控。
    • 娱情监控，抓取微博、各论坛、公司相关关键字，有没有投诉之类的。

    基于Zabbix监控体系总结

    • 用Zabbix监控硬件、系统健康；
    • 通过Zabbix自定义key监控应用服务，比如数据库主从状态也加到Zabbix上；
    • 网络监控，流量分析用piwik或百度统计；
    • jvm监控；
    • 自动化监控；
    • 日志监控使用ELK；
    • 安全监控Nginx+lua web应用防火墙；
    • 业务监控，Zabbix自定义key，或者写一些接口能够获取到，或者开发专门的业务监控系统；
    • 业务波动可以通过Zabbix自定义的screen基本能找到问题的根源，再通过ELK日志监控，加上流量分析，能找到问题的根源。
